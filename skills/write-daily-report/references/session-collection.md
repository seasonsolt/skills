# 当日 Codex / Claude Code / Pi Agent 会话采集与工作过滤

仅在用户要求从当日活跃的 Codex、Claude Code、Pi Agent 任务或 session 自动生成日报时读取本文件。

## 先确定工作范围

会话正文可能包含个人信息和无关项目。先确定一个或多个批准的工作根目录，再读取正文。

工作根目录按以下优先级确定：

1. 用户在本次请求中明确指定的目录。可以指定多个目录，也可以明确替换已有配置。
2. 环境变量 `DAILY_REPORT_WORK_ROOTS`。多个目录使用当前操作系统的路径分隔符分开：macOS/Linux 通常为 `:`，Windows 通常为 `;`。
3. 当前任务明确位于一个工作 Git 仓库时，使用 `git rev-parse --show-toplevel` 得到的源仓库根目录。只有在当前仓库显然属于本次汇报范围时才使用此回退。
4. 仍无法确定时，不读取任何会话正文；只问用户一个简短问题，例如“本次日报应纳入哪些工作目录？”

将 `~` 展开并解析真实路径。满足以下任一条件的会话进入目录白名单：

1. 实际 `cwd` 位于任一批准工作根目录或其子目录。
2. Codex 会话的实际 `cwd` 位于 Codex worktree 根目录，并且通过 Git common-dir 解析出的源仓库位于任一批准工作根目录或其子目录。

范围安全规则：

- 不要默认扫描用户主目录、磁盘根目录或包含大量无关仓库的宽泛父目录。
- Codex worktree 不能只因实际 `cwd` 位于 `~/.codex/worktrees` 而纳入。使用 `scripts/resolve_codex_session_scope.py` 调用 `git rev-parse --git-common-dir`，以源仓库真实路径决定范围。
- 普通临时目录或其他工作树不享受此例外；不要根据目录名称、remote URL 或会话正文猜测来源。
- 不要把 Claude 项目目录的编码名称反解为路径来猜测归属，始终以会话记录中的实际 `cwd` 和规范化路径为准。
- 三个脚本都接受可重复的 `--work-root`；未传参时读取 `DAILY_REPORT_WORK_ROOTS`，两者都没有时会返回错误而不是使用个人默认值。

## 采集流程

1. 按用户当前时区计算目标日期 `00:00:00` 至次日 `00:00:00`。同时检查当前环境中可用的 Codex、Claude Code 与 Pi Agent 来源；某个来源不存在时继续处理其他来源。
2. 先按规范化 `cwd` 执行目录白名单；对 Codex worktree 额外验证 Git common-dir。未通过任一条件的会话不读取正文；Pi Agent 必须先仅读 session header 中的真实 `cwd`，通过后才读取消息。
3. 排除当前正在生成日报的任务，避免把“生成日报”写成工作成果。
4. 对白名单内候选读取最近状态与回合摘要。需要确认累计结果或交接关系时再向前翻页；不要加载附件、思考过程、大段命令输出或完整 tool result。
5. 从用户消息、最终答复、状态更新、测试结果、文件变更、提交或推送记录中提取事实。推理摘要和过程性工具输出不能单独作为工作成果。
6. 识别委派来源、同名续接任务、相同工作目录、分支、提交、交付物和连续指标基线，合并为一条成果链。
7. 对跨 Codex、Claude Code 与 Pi Agent 的同一工作再次去重；不能因为客户端不同而重复累计成果。

## Codex 采集

1. 使用当前环境可用的 Codex 任务或线程列表能力读取近期任务；若没有该能力，跳过 Codex 并记录覆盖限制。
2. 对目标日期更新的任务，先按唯一 `cwd` 批量运行范围解析器：

   ```bash
   python3 scripts/resolve_codex_session_scope.py \
     --work-root /path/to/company-projects \
     --work-root /path/to/client-project \
     --cwd /path/from/thread \
     --cwd /another/path/from/thread
   ```

3. 仅对结果中 `included=true` 的任务加载内容。`mode=direct` 表示 cwd 直接命中白名单；`mode=codex_worktree` 表示已从 Git common-dir 验证源仓库。
4. 主任务、委派任务和续接任务优先按来源关系合并；没有显式来源时按源仓库、目录、分支、文件、提交和指标链判断。

## Claude Code 采集

从技能目录运行自带脚本，日期使用用户当前时区的本地日期：

```bash
python3 scripts/collect_claude_sessions.py \
  --date YYYY-MM-DD \
  --work-root /path/to/company-projects \
  --work-root /path/to/client-project
```

- 脚本读取 `${CLAUDE_CONFIG_DIR:-~/.claude}/projects/**/*.jsonl`，忽略 `subagents/`，依据每条记录的时间戳判断当日活动，并依据记录中的真实 `cwd` 执行白名单。
- 只消费脚本输出中的用户或助手文本摘要、标题、目录、分支和时间；脚本不读取 tool-results、附件、思考块或子代理 transcript。
- 会话文件可能跨日追加，因此文件当日被修改不等于全部内容属于当日；只能使用目标日期时间窗内的消息作为事实。
- Claude 配置目录不存在或没有白名单内会话时按零条处理，不阻断日报。

## Pi Agent 采集

从技能目录运行自带脚本：

```bash
python3 scripts/collect_pi_sessions.py \
  --date YYYY-MM-DD \
  --work-root /path/to/company-projects \
  --work-root /path/to/client-project
```

- 脚本默认读取 `${PI_CODING_AGENT_SESSION_DIR:-${PI_CODING_AGENT_DIR:-~/.pi/agent}/sessions}` 中的顶层 session JSONL；兼容 `--pi-home` 和 `--session-dir` 显式覆盖。
- Pi Agent session 首行 header 提供真实 `cwd`。脚本先按该字段执行白名单，未通过时不读取后续消息；不要根据编码后的目录名猜测项目归属。
- Pi Agent session 是追加式消息树。只消费当前 leaf 的有效分支，避免把已放弃分支的旧结论写入日报；v1 线性 session 按文件顺序读取。
- 只消费目标日期时间窗内 user/assistant 的 text block；忽略 thinking、tool call、tool result、附件、custom entry 和原始命令输出。
- 默认不扫描 Pi subagents 在父 session 目录下保存的嵌套 transcript，防止与主 session 重复；需要累计结果时以主 session 中可验证的最终文本为准。
- 使用 `session_info.name` 作为可选标题、`parentSession` 辅助识别 fork 或续接关系。Pi 配置目录不存在或没有白名单内 session 时按零条处理，不阻断日报。

## 工作相关性判定

目录白名单决定“可以读取”，岗位和汇报目标决定“应该写入”。判断标准不是是否花了时间，而是内容是否服务于用户需要向组织、客户、团队或项目负责人汇报的目标和交付。

优先级从高到低：

1. 用户本次明确指定的岗位、项目、目录和纳入或排除范围。
2. 会话是否在批准目录内产生与岗位目标直接相关的代码、内容、设计、测试、数据、运营结果、业务结论、协作结果或其他交付物。
3. 下方默认排除类别，用于处理白名单内仍与本次汇报无关的元会话。

不要使用职业刻板规则。例如内容、设计、营销或 AI 工具建设可能正是用户的岗位交付；只要用户范围和证据支持，就应纳入。

## 默认纳入

- 需求分析、方案设计、代码或内容生产、测试、缺陷处理、发布、运营和运行保障等岗位交付。
- 形成明确结论或交付物的调研、评审、架构评估、安全审计、数据分析和用户研究。
- 与目标进度直接相关的计划、风险、决策、跨团队协作和待办。
- 为组织或项目交付而建设的测试、CI/CD、质量门禁、自动化、知识库或工程平台能力。

## 默认排除

除非用户明确说明属于岗位职责或本次汇报范围，否则不进入日报：

- 当前日报生成、聊天整理、任务归档或重命名等元操作。
- 与批准项目无关的生活、娱乐、个人事务和个人内容。
- 没有形成项目决策或交付物的随手问答、内容消费和工具探索。
- 仅改善个人 AI 使用体验的全局配置或试验；若 AI 工具、skill 或自动化本身就是岗位交付，则应纳入。
- 已放弃分支、被后续结论推翻的中间结果和仅有过程、没有可验证进展的噪声。

## 混合任务、去重与状态

- 同一会话包含工作交付和个人操作时，只记录工作范围内的交付及其验证结果。
- 同一会话先调研通用工具、后形成项目实施计划时，只记录被项目采用的结论、计划和交付物。
- 优先使用最新、证据最完整的回合描述累计成果；早期中间值只用于计算可信增量。
- 不要把上游主任务和下游委派任务的同一批测试、同一提交或同一文档分别计数。
- 不同客户端若出现相同提交哈希、文件集合、测试批次或首尾一致的指标链，视为同一成果；合并上下文但指标只计一次。
- 仍在运行的任务写为“进行中”，只列已经验证的阶段结果；不要推测最终状态。
- 跨日任务可以纳入目标日期实际发生的新进展，但不要把历史累计成果冒充为当日新增。无法拆分增量时写“截至当日累计”，并明确口径。

## 隐私与输出

- 不在日报中复制 API key、token、密码、Cookie、私密对话、会话文件路径或完整命令输出。
- 默认只输出日报正文，不列出被排除会话或本地目录。
- 用户要求审计时，使用简短表格列出任务名称、判定（纳入/排除/混合）和一句原因，不泄露会话正文。
- 自动采集且来源不完整时，在正文末尾简要说明实际检查了哪些客户端；对不可用或没有记录的来源使用中性事实，不暗示已完整扫描。
