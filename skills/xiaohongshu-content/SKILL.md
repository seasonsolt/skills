# 小红书内容创作到发布全流程

这是一条被真实跑通过的链路：**诊断 → 定位 → 选题 → 文案 → 配图 → 草稿 → 发布**。
它把"运营直觉"拆成可复用的动作。核心信念：**每一步都用真实数据/真实用户声音说话。**

## 什么时候用哪几步

用户很少一次要走完全程。先判断他卡在哪，只做需要的那几步：

- "我小红书数据不好 / 低转化" → 从 **1 诊断** 开始，定位漏斗是哪一环漏水，再对症做后续步骤。
- "帮我改简介 / 主页没人关注" → **2 定位**。
- "帮我想个选题 / 这个话题怎么写" → **3 选题 + 4 文案**，并给出卡片结构。
- "把这篇文章/播客/框架/热点消息做成小红书" → **3→5**；若用户要草稿或发布，继续 **6→7**。
- "帮我做图 / 确认 / 继续 / 生成卡片" → **5 配图**，把已确认卡面文案渲染成图，不要停在文字稿。
- "帮我建草稿 / 发布 / 用 Chrome" → **5→7**；先确保图片已生成，再进浏览器建草稿或报告明确阻塞点。

症状→环节→归因的完整漏斗表在 `references/diagnose.md`（单一真源），诊断时读它。

## 默认推进规则（防止半成品）

- 除非用户明确说"只要正文 / 只写文章 / 不要卡片"，小红书内容交付应包含：标题、正文、标签和一套完整卡片脚本。卡片数量由内容密度决定，通常 5–10 张，不为凑数拆页，也不把多个核心观点硬塞进一页。
- 不要在只产出正文后停止；正文完成后继续给出卡片结构和卡面文案。
- 做知识 / 框架 / 技术概念 / 系统机制类卡片时，先做一次 **infographic 触发预检**（触发条件与变量表见 5 配图 → `references/image-cards.md`）：触发就在卡片脚本里预留 1 张"复杂概念总览图"，不触发也在归档里写一句原因。
- 卡面文案尚未确认时，先输出完整的 P1→Pn 卡片脚本并说明总张数与拆分依据，再请求确认；用户说"确认 / 可以 / 继续 / 生成 / 做图"即视为逐字确认。
- 卡面文案已确认后，必须进入 **5 图文卡片** 生成或渲染图片。P1 封面是点击入口，默认必须用 image2 / 图像模型绘制高冲击视觉封面；中文标题可用 HTML/CSS 叠字保证准确。品牌 / 产品 / 人物新闻封面还必须通过“去标题识别测试”：隐藏标题后，仍能凭官方标识、品牌视觉、产品界面或标志性主体认出主题；只有通用科技感、时钟、箭头等隐喻不算通过。若仓库存在 `assets/card-template/`，P2 之后的文字密集内容优先用它做确定性渲染，除非用户明确要求全套 AI 出图。
- 用户要求草稿、发布，或显式要求 Chrome/浏览器时，不要只解释原因；先做可做的前置工作（归档、渲染图片、准备正文/标签），再执行浏览器预检和草稿尝试。
- 每次进入下一阶段时更新私有归档：`posts/<YYYY-MM-DD>-<slug>/note.md` 和 `cards/`。`posts/` 属私人内容，**绝不进任何公开仓库**（是否提交到你自己的私有仓由你决定）。

## 归档约定（建议）

把每篇笔记归档到项目里，形成"发布→复盘→下一篇"闭环——下次诊断（第 1 步）能直接读历史。

- **每次开工先读 `posts/PLAYBOOK.md`（若存在）**：账号级实测打法（流量结构、双引擎、硬规则），优先级高于通用直觉；它是私有的，随 `posts/` 一起被 gitignore。
- 在 `posts/<YYYY-MM-DD>-<slug>/` 新建目录，写 `note.md`（字段见 `examples/note.template.md`：title/slug/date/status/format/series/source/tags/metrics + 正文 + 卡片表），卡片图放 `cards/`（P1…Pn 命名）。
- 发布后把真实数据回填进 `note.md` 的 `metrics`，写一句复盘。
- `posts/` 属于**使用者的私人内容**，默认不进公开仓库（`.gitignore` 已排除）；本仓库只提供 `examples/` 里的模板。

## 工作流

### 1. 数据诊断 → `references/diagnose.md`
登录 `creator.xiaohongshu.com`，拉近 7 日 / 近 30 日「笔记数据总览」+「笔记管理」逐篇数据，
算出曝光→点击→阅读→互动→涨粉的漏斗，找出最弱的一环。对比表现最好和最差的笔记，找规律
（图文 vs 视频？哪类选题？受众是"点赞型"还是"收藏型"？）。**先定位问题，再动手，别一上来就改。**
**完成判据**：给出"你卡在 X 环节，因为 Y"一句话结论 + 2-3 个具体动作。

### 2. 人设定位 → `references/positioning.md`
简介三行：**人设**（你是谁）/ **定位**（写什么，点名可被搜索的关键词）/ **关注理由**（更新承诺 + 行动引导）。
⚠️ 简介只能在**手机 App** 改，网页版没入口——把文案写好让用户去粘贴。
**完成判据**：产出可直接粘贴的三行简介，每行各司其职、承诺能兑现。

### 3. 非共识选题 → `references/non-consensus.md`
**别写共识和常识**（"各有所长""结论前置"这种全网烂大街的角度没人看）。
全网检索真实用户的争议、吐槽、反直觉数据（Reddit / V2EX / 知乎 / HN），每个观点带**具体出处 + 数字**。
争议本身能拉互动。
按 `references/non-consensus.md` 的混合路由选择 Tavily / Codex-WebSearch；关键数字必须双源核验，不能把检索器生成的摘要当出处。
**完成判据**：至少 3 个非共识点，每个都挂了具体出处 + 数字；每个关键数字至少 2 个独立来源且含 1 个一手来源；纯共识观点一个不留。

### 4. 结构化文案 → `references/copywriting.md`
钩子开头（前两行 = 信息流摘要，必须命中痛点）+ 结论前置 + 分步骤 + **系列化 + 关注引导**结尾。
标题 ≤20 字。核心杠杆是把单篇变成"追更对象"，并在收藏动作前拦一句关注请求。
**完成判据**：标题 ≤20 字；正文含钩子开头 + 四个转化机制（系列化/下期预告/评论引导/收藏拦截）中至少 3 个；**正文至少设计 1 处接话空间且过"非本领域者一句话能接"门槛护栏**（见 `references/接话空间.md`）；**且正文已过 `humanizer-zh` 去 AI 味、≥40/50**（见红线「去 AI 味强制化」）。

### 5. 图文卡片 → `references/image-cards.md`
知识 / 框架类内容用**图文 carousel（不是视频）**——框架类收藏率最高。
卡片结构按内容动态拆分：封面 + 可选痛点/背景 + N 个知识点 + 可选总览图 + 收尾，通常 5–10 张。每张只承担一个清晰任务；信息少就少做，信息多则合理拆页，不得为了固定张数注水或压缩。P1 封面必须用 image2 / 图像模型绘制高冲击视觉，目标是吸引点击进入；中文标题可后期叠加。热点解读、框架拆解、文字密集卡默认用仓库 `assets/card-template/` 做 **方案 B（HTML/CSS 确定性渲染）**，保证中文可控；Codex、Claude Code 或其他能编辑文件并运行本地渲染器的 agent 都能执行。P2 之后只有在用户明确要求真实视觉/插画/AI 出图，或该页需要复杂无字母图时，才继续用 **方案 A**。转载他人内容必标出处。
复杂技术概念必须执行 **infographic 触发预检**（触发判据、变量与母提示词见 `references/image-cards.md` 与 `references/infographic-diagram-prompt.md`）：触发后生成 1 张无正文横向信息图，作为方案 A 的专用分支，不替代 3:4 卡片与 HTML 中文排版。
**先决条件**：全部卡面文字稿先经**用户逐字校验定稿**，才开始生成图——别拿没定稿的文案去烧图，改字等于重画。
**完成判据**：归档里有 infographic 触发判断；若触发，卡片脚本包含总览图页和已填写的 infographic 变量。全套图已落到 `posts/<...>/cards/`，核对张数、P1→Pn 顺序、1080×1440 尺寸；品牌 / 产品 / 人物新闻封面已通过“去标题识别测试”；并至少抽查封面和文字最多的 1-2 张，修复断行、遮挡、错字或主题识别不足后重新渲染。

### 6. 建草稿 → `references/draft.md`
`creator.xiaohongshu.com/publish/publish?target=image`。图文流程必须先有图才能进编辑器；
没图可用「文字配图」生成占位。填标题 / 正文 / 用 #话题选择器挑高流量标签。存草稿点「暂存离开」。
用户显式要求 Chrome/浏览器/现有登录态时，先使用对应浏览器能力做预检；若不可用，报告具体阻塞点（例如扩展未安装、未启用、选中 profile 不对、未登录），不要静默换后端。
**完成判据**：草稿箱里出现该笔记，标题/正文/标签齐全；若图片上传失败，明确标记"草稿未完成"，并给用户最短手动补图步骤。

### 7. 发布 → `references/publish.md`
上传图片、删占位、核对顺序、发布。⚠️ **浏览器 `file_upload` 有安全白名单**，多数环境下自动上传图片会被拒，
最后一步大概率要用户手动拖图。发布后**回复所有评论**——是给算法的活跃信号，二次推流。
**完成判据**：全部卡片按 P1→Pn 在草稿内、占位已删、顺序已核对；发布后回填 `note.md` 的 metrics。

## 贯穿全程的红线

- **合规优先（最高优先级）**：小红书**明令禁止**这类内容——展示/交易境外软件与网站、提供境外软件账号、教翻墙/下载境外软件、跨区礼品卡充值教程。这类内容即便会火也别做——真实案例中，有账号因发《订阅教程（美区 + 礼品卡）》被**直接下架、不可被他人查看**，并被警告"多次违规加重处罚"。同题材要改成合规角度：只讲工具怎么用、怎么选，不教怎么搞到境外账号。选题阶段（第 3 步）就先过这一关。
- **数据说话**：诊断结论、选题角度、用法举例，都要有真实依据；不确定的数字（如订阅价格）发布前核实，别被评论区打脸。
- **真实性**：文案里的"我的用法""亲测"必须是真的；转载内容标清出处，别包装成原创。这是"真从业者"人设的根基。
- **🔴 去 AI 味强制化（写任何要发布的正文必做，默认执行、不需用户点名、未过不算完成）**：正文定稿前必须过一道 humanizer-zh，按 24 项 AI 痕迹改写 + 50 分 rubric 打分，**≥40/50 才算完成**；仅当它不可用（未安装/报错）才 fallback 到 DeepSeek 母语重写。去味时**事实 / 数字 / 出处一律不许改，不许编造第一人称经历**（与「真实性」同级）。调用方式（Claude Code 用 `/humanizer-zh`；Codex 等无斜杠命令的 agent 读并套用 humanizer-zh 的方法）与完整流程见 `references/copywriting.md`，工具全表见 `references/去AI味-工具与提示词.md`；跨 agent 治理另见仓库根 `AGENTS.md`。
- **不绕安全边界**：`file_upload` 传不了就老实交回用户手动，**绝不把文件塞进扩展内部目录去骗过校验**——那是绕过安全控制，会被拦，也不该做。
- **改已发布的笔记会重新审核**，有限流风险，尤其正文含外链 / 礼品卡时。要改就做增量修改（只加头尾），别大动。
