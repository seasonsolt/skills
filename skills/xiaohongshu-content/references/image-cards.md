# 图文卡片

## 为什么是图文，不是视频

知识/框架/清单类内容，**用图文 carousel（多图翻页），不要做视频**。理由：
1. 框架类内容"可收藏"，而收藏是小红书知识类的核心行为，图文收藏率最高。
2. 一个知识点一张卡，翻页节奏天然成立。
3. 搬别人的视频会被判"非原创"限流；图文用中文重讲是二次创作，算原创、更安全。
4. **先看该账号自己的数据**：多数技术/知识账号图文显著优于视频。用户数据 > 通用经验。

## 卡片结构（数量随内容变化）

通常做 5–10 张，但这不是配额。先列出必须讲清的内容单元，再决定总张数：封面和收尾各占一张；痛点、背景和总览图只在确有必要时加入；核心部分坚持一卡一个要点。信息少就少做，信息多就拆页，不为凑数制造重复页，也不为压缩张数堆字。

```
P1 封面    —— image2 / 图像模型绘制高冲击视觉封面，中文标题可用 HTML/CSS 叠加
P2 痛点    —— 让读者对号入座（"你是不是也这样👇"），引出解法
P3..Pn 知识点 —— 一卡一个要点，标题带序号
P(末) 收尾  —— 一句话总结 + 关注引导 + 出处
```

每张卡写两块：
- **卡面大字**：真正放到图上的文字，短行、一点一卡、留白多。
- **设计备注**：配色/强调色/图标/排版提示（给做图的人或图像模型看）。

## P1 封面硬规则

P1 是信息流点击入口，必须使用 image2 / 图像模型绘制，不用纯 HTML 文字卡替代。封面目标不是完整讲清知识点，而是制造停留和点击：强主体、强对比、强情绪、缩略图也能看出主题。

执行方式：
1. 先用图像模型生成 P1 视觉底图或完整封面；优先做无字或少字图，避免模型写错中文。
2. 若标题必须精确，使用 HTML/CSS 在 image2 底图上叠加中文标题、短副标题和系列标识。
3. P1 最终仍需输出 1080×1440，文件名按 `P1-*.png`。
4. 归档的制图记录里写明：P1 使用 image2 生成，若有叠字也写明叠字方式。

### 品牌 / 产品 / 人物新闻：去标题识别测试

新闻封面不能只把“延期、发布、涨价、下架”等事件画出来，还必须让人一眼知道**是谁 / 什么产品发生了这件事**。生成或合成 P1 后，临时隐藏全部标题、副标题和标签，再做一次识别测试：

1. 缩到信息流缩略图尺寸，只看画面，不读文字。
2. 画面至少包含 **2 个相互独立的主题信号**，且至少 1 个是主体专属信号：官方标识、准确的品牌视觉、真实产品界面、产品外形、人物肖像或其他标志性主体。
3. 通用的科技粒子、服务器、时钟、日历、箭头、手机或人物背影只能表达事件，不能单独充当主题信号。
4. 若换掉标题后，同一张图还能套给多个品牌或产品，判定为不通过，必须补充主体信号或重画。

品牌准确性优先于“像某品牌”的氛围：

- Logo、产品名、模型名和界面文字优先取自官方素材或可信的一手页面，再用 HTML/CSS 或确定性合成叠加；不要让图像模型凭记忆伪造 Logo 或拼写。
- 品牌色只能作为辅助信号，不能代替官方标识、产品界面或标志性主体。
- 没有可靠官方素材时，不编造标识；改用真实产品界面、明确的产品形态或其他可核验元素，并在归档里注明素材来源。
- 最终抽查同时看两版：完整封面负责点击率，隐藏标题版负责主题识别。两版都通过才算完成。

## 生成方案：按内容选 A/B

- **方案 A：图像模型生成** —— P1 封面默认使用；用户要求真实视觉、插画、照片感、强审美时也用。优先调用当前 agent 环境已有的图像生成工具；没有原生工具时，再使用可用的浏览器会话或图像 API。
- **方案 B：HTML/CSS 确定性渲染** —— 文字密集、热点解读、框架拆解默认用。Codex、Claude Code 或其他能编辑文件并运行本地渲染器的 agent 都可执行；中文像素级精确、零错字、完全可控，缺点是要自己复刻系列视觉。
- **复杂技术概念的横向总览图** —— 先执行下方 `infographic-diagram` 触发预检；触发后走方案 A 的专用分支，使用 [`infographic-diagram-prompt.md`](infographic-diagram-prompt.md)。这类图负责呈现系统结构与关系，不承载正文；正式中文仍由 HTML/CSS 叠加。
- **切换判据**：A 连续失败 2-3 次（含重试）就切 B，别在 A 上死磕浪费时间。
- **推进判据**：用户说"确认 / 可以 / 继续 / 生成 / 做图"后，视为卡面文字已定稿，必须进入出图或渲染，不要继续只给文字方案。

## 方案 A：用图像模型生成

先检查当前环境是否提供原生图像生成或编辑工具。Codex 若有 `imagegen`，直接调用并把结果落到当前笔记的 `cards/`；不要为了生图绕到 ChatGPT 浏览器。只有当前环境没有原生工具时，才使用已有登录态的 ChatGPT 浏览器或图像 API。

流程：
1. **先只画封面 P1**，验证构图、风格和中文渲染质量；不合格时不要盲目生成整套。
2. 原生工具支持参考图或编辑时，把已确认的上一张成图作为风格参考，逐张生成 P2…Pn；不要每张从零猜风格。
3. 浏览器后端需要在同一个对话里迭代，逐张说明“沿用相同的视觉系统”；API 或原生工具则显式传入参考图。
4. 少量必须由模型绘制的中文放在引号里，并明确“原样清晰渲染、不要错字、不要多字”。正文或排版敏感内容改用无字底图 + 方案 B。
5. 每张结果直接保存到 `posts/<...>/cards/`，并按完成判据检查文件、尺寸和顺序。

### infographic-diagram：复杂技术概念总览

知识 / 框架 / 技术概念 / 系统机制类内容，先做这张预检表。只要命中任一条，就必须在 P1→Pn 脚本里预留 1 张"复杂概念总览图"，并读取 `infographic-diagram-prompt.md` 填变量；这张图计入根据内容确定的总张数。

| 触发项 | 判断标准 | 命中示例 |
|---|---|---|
| 组成部分多 | 同一主题要解释 3 个以上核心组件 | memory、tools、loop、eval、ops |
| 关系复杂 | 组件之间存在流程、依赖、层级、对照任一关系 | 输入到输出的系统链路 |
| 有反馈回路 | 结果会反过来修改前面的配置、策略或数据 | eval 结果回写 prompt / retrieval |
| 读者需要总览 | 单页术语解释会让读者看懂局部，但看不懂整体结构 | agent harness、LLM Ops、RAG pipeline |

不触发的情况也要写明原因，例如："仅是 2 个工具对比，无流程或反馈回路，继续用普通卡片。"

触发后先做关系拆解，再套用专用模板：

1. `SUBJECT`：用一句话写清要解释的系统或机制，不写泛主题。
2. `NUMBER`：只统计上方承担主要叙事的“大型科学形式”，通常 3–7 个；不要把装饰图标算进去。
3. `CONTENT CATEGORIES`：把下方模块分成互斥且同一层级的类别，例如“输入 / 调度 / 执行 / 反馈”。
4. 明确关系类型：流程、依赖、反馈、对照或层级；让连接线有语义，不做随机装饰。
5. 有参考图时只提取材质、线条、色彩、密度和图解气质，禁止复刻参考图的对象、图标、布局或内容。
6. 先生成无可读正文的横向母图。需要用于小红书时，再把母图裁切或嵌入 3:4 卡片，并用 HTML/CSS 添加经过确认的中文。

归档要求：在 `note.md` 的卡片区前写 `infographic_check`，包含 `triggered`、`reason`、`SUBJECT`、`NUMBER`、`CONTENT CATEGORIES`、`RELATIONSHIP`。若未触发，`reason` 必须说明为什么普通卡片足够。

若概念只有一条简单流程或 1–2 个对象，且没有反馈回路，继续使用普通卡片模板，不要为了“显得复杂”启用此分支。

## 方案 B：HTML/CSS 确定性渲染

文字密集内容默认用这套；方案 A 出现错字、精确布局失败或环境没有图像工具时也切到这套。它是与 agent 无关的本地渲染流程：Codex、Claude Code 或其他 agent 均可编写卡片数据与样式、运行渲染器并检查成图，不依赖 ChatGPT。
**现成模板**：`assets/card-template/`（含 CSS 视觉系统 + `build_cards.py`）——复制一份、改 CARDS 里的中文即可，别从零写渲染器。手动做法：

1. 写一个自包含 HTML：按已确认的实际张数生成 P1→Pn，每张卡对应一个 `.card` div，尺寸固定 3:4（如 1080×1440），内联 CSS 复刻系列视觉——浅米色底、左上引号装饰、大字标题、红/黄强调 `<span>`、大量留白。中文直接写进 HTML（**像素级精确、零错字**，这是 B 相对 A 的最大优势）。
2. 渲染截图（二选一）：
   - **本地无头浏览器（优先）**：直接运行模板自带的 `build_cards.py`，让 Chrome/Chromium 渲染并截图，不需要浏览器 MCP。
   - **本地文件 + 浏览器 MCP**：没有可用的无头浏览器脚本时，把 HTML 存到 `posts/<...>/cards/cards.html`，浏览器打开，对每个 `.card` 逐个 `scroll_to` + 截图；或用 JS 把每个 `.card` 用 `html2canvas`/`canvas` 导出 PNG 下载。
   - 视口设成卡片尺寸，逐张截，存成 `P1-*.png … Pn-*.png`。
3. 存进 `posts/<...>/cards/`，核对张数与顺序。
4. 必须检查每张图尺寸为 1080×1440；至少打开封面和文字最多的 1-2 张检查断行、遮挡、错字，发现问题就改 `build_cards.py` 后重跑。

视觉尽量贴近上一篇的系列风（浅米卡片 + 引号 + 红/黄强调），保持系列识别度。

## 把图落地（方案 A 的浏览器后端专用）

Codex 原生 imagegen 或 API 结果应直接保存到 `cards/`，不走这一节。仅从 ChatGPT 浏览器取图时：图片加载后，用页面提供的下载能力保存原图；若连续下载被 Chrome 拦截，逐张下载，并用 `ls` 核对数量。

⚠️ 别用 `canvas.toDataURL` 把 1MB+ 的 base64 塞回对话——会把上下文撑爆。走"下载文件"这条路。

## 红线

- **转载他人内容必标出处**（如"演讲：某某 @ 某大会，我做了中文提炼"+ 源码链接）。既尊重版权，也是信任背书，别包装成自己原创。
- 提示词里替读者"落地化"的举例（如某术语的类比），发布前让用户换成他**真实用过**的例子，更有说服力。
